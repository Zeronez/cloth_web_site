# Incident Response Runbook

Цель: быстро локализовать, восстановить и документировать инцидент.

## Роли
- Incident Commander (IC): координация.
- Ops: инфраструктура/деплой.
- Backend: API/DB/Celery.
- Frontend: клиент/SSR.
- Support: коммуникации с пользователями (если нужно).

## Severity (пример)
- SEV1: checkout/payments не работают, массовые 5xx, утечка данных.
- SEV2: деградация p95, частичные ошибки, проблемы с одной функцией.
- SEV3: минорные баги без влияния на продажи.

## Процесс
1) Зафиксировать время начала, симптомы, затронутые компоненты.
2) Остановить кровотечение (rollback, feature flag, отключить провайдера).
3) Восстановить сервис.
4) Проверить целостность данных (заказы/оплаты/склад).
5) Коммуникация (внутри команды + пользователям при необходимости).
6) Пост‑мортем: причины, действия, prevention.

## Быстрые проверки
- Backend: `/health/live/`, `/health/ready/`
- Логи ошибок (5xx) + всплески по роутам
- DB: connections/locks/slow queries
- Celery: очередь растёт? воркеры живы?
- Платежи: вебхуки принимаются? signature ok?

## Артефакты пост‑мортема
- таймлайн
- root cause
- impact (заказы/выручка)
- action items с владельцами/датами
