# Диаграммы для ВКР: AnimeAttire (интернет‑магазин аниме‑стритвира)

Формат: для каждой диаграммы указаны **цель**, **что должно быть на диаграмме**, **акторы/сущности**. Этот список можно использовать как чек‑лист при подготовке иллюстраций и вставлять их в текст как плейсхолдеры вида: `[фото № - описание фото]`.

## 1) Use Case Diagram (UML) — «Варианты использования»
**Цель:** показать, кто и какие функции системы использует (пользовательские требования).  
**Элементы:** акторы, варианты использования, include/extend, граница системы.  
**Акторы:** Гость, Покупатель, Администратор/Контент‑менеджер, Менеджер заказов (опционально), Платёжный провайдер (внешний), Служба доставки (внешняя), Email/SMS сервис (опционально).

**Рекомендуемые use cases:**
- Просмотр каталога / фильтры / сортировка / поиск
- Просмотр карточки товара
- Регистрация / вход / выход / восстановление пароля
- Управление профилем и адресами
- Работа с корзиной (add/update/remove)
- Оформление заказа (checkout)
- Оплата заказа (создание платёжной сессии / подтверждение)
- Просмотр статусов и истории заказов
- Избранное
- Обращение в поддержку
- Админ: CRUD категорий/франшиз/товаров/вариантов/изображений
- Админ: управление заказами и статусами

## 2) User Flow / Customer Journey — «Пользовательский путь»
**Цель:** описать UX‑сценарий без технических деталей.  
**Элементы:** шаги/экраны, развилки, точки ошибок, альтернативные пути.  
**Потоки:**
- Гость → каталог/поиск → карточка → корзина → авторизация (опционально) → checkout → оплата → успех/ошибка
- Покупатель → история заказов → повторная покупка (опционально)
- Покупатель → отмена/возврат (если предусмотрено)

## 3) BPMN 2.0 — «Бизнес‑процесс обработки заказа»
**Цель:** формализовать процесс от создания заказа до закрытия.  
**Элементы:** пулы/дорожки, задачи, события, шлюзы, сообщения, данные.  
**Дорожки (lanes):** Покупатель, Система, Платёжный провайдер, Склад/Менеджер заказов, Служба доставки, Email/SMS (опционально).

**Ключевые ветки:**
- Создание заказа → резервирование товара (если есть) → запрос оплаты → подтверждение/отказ
- Комплектация → передача в доставку → смена статусов → уведомления → закрытие
- Отмена до оплаты / отмена после оплаты (возврат)

## 4) DFD — контекстная (Level 0) и декомпозиция (Level 1)
**Цель:** показать потоки данных между внешними сущностями, процессами и хранилищами.  
**Элементы:** процессы, хранилища, внешние сущности, потоки данных.

**Level 0:** один процесс «Интернет‑магазин» + потоки: Каталог/Заказы/Оплата/Доставка/Уведомления.  
**Level 1 (пример декомпозиции):**
- Управление пользователями/сессиями
- Каталог и поиск
- Корзина
- Оформление заказа
- Платежи и webhooks
- Доставка и трекинг
- Админ‑управление каталогом

## 5) ERD — «Модель данных»
**Цель:** зафиксировать структуру БД и связи сущностей.  
**Элементы:** сущности/таблицы, PK/FK, кардинальности, уникальности.

**Рекомендуемый минимум сущностей:**
- `User`, `Address`
- `Category`, `AnimeFranchise`
- `Product`, `ProductVariant` (SKU), `ProductImage`
- `Cart`, `CartItem`
- `Order`, `OrderItem`
- `DeliveryMethod`, (опционально) `DeliveryTrackingEvent`
- `PaymentMethod`, `Payment`, (опционально) `PaymentEvent`/`Refund`
- `FavoriteProduct`
- `ContactRequest`

## 6) UML Sequence Diagrams — «Последовательности» (3–5 сценариев)
**Цель:** показать обмен сообщениями между фронтендом, API и внешними сервисами.  
**Элементы:** lifelines, сообщения, alt/opt, ответы/ошибки.

**Рекомендуемые сценарии:**
- Авторизация/регистрация (+ синхронизация корзины гостя)
- Добавление товара в корзину
- Checkout: создание заказа → создание платежа → webhook → обновление статусов
- Просмотр заказа и трекинга доставки
- Админ: создание/редактирование товара и загрузка изображений

## 7) UML Component Diagram — «Компоненты системы»
**Цель:** показать состав подсистем и зависимости.  
**Элементы:** компоненты, интерфейсы, зависимости.

**Рекомендуемые компоненты:**
- Frontend (Next.js): Catalog UI, Product UI, Cart/Checkout UI, Account UI
- Backend (Django/DRF): Auth, Catalog, Cart, Orders, Payments, Delivery, Support
- DB (PostgreSQL), Redis/Celery (если используется)
- Storage (медиа), Payment provider, Delivery provider

## 8) UML Deployment Diagram — «Развертывание»
**Цель:** показать, где физически развернуты части системы и по каким протоколам общаются.  
**Элементы:** nodes/containers, артефакты, связи.

**Узлы:**
- Browser
- Frontend (SSR/Static)
- Backend API
- PostgreSQL
- Redis/Celery (опционально)
- Object storage/CDN (для изображений)
- External: Payment, Delivery

## 9) State Machine Diagram — «Состояния заказа»
**Цель:** формально описать жизненный цикл заказа.  
**Состояния (пример):** `Created` → `AwaitingPayment` → `Paid` → `Processing` → `Shipped` → `Delivered` → `Closed`, ветки `Cancelled`, `PaymentFailed`, `Refunded`.

## 10) Activity Diagram — «Алгоритм оформления заказа»
**Цель:** показать логику сценария «покупка» с ветвлениями.  
**Включить:** проверка наличия, выбор доставки/оплаты, успех/ошибка оплаты, финал.

## 11) Sitemap / Навигация сайта (опционально)
**Цель:** структура страниц и маршрутов.  
**Включить:** Главная, Каталог, Карточка, Корзина, Checkout, Профиль, Заказы, Избранное, Контакты, Доставка, Возврат, Админ (если есть).
