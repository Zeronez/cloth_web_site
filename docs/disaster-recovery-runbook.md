# Disaster Recovery Runbook

Цель: восстановить работоспособность после серьёзного инцидента (падение VM, потеря диска, ошибка миграции, компрометация).

## Приоритеты
1) Безопасность (остановить утечку/компрометацию).
2) Восстановить checkout/приём заказов.
3) Восстановить каталог и кабинет.
4) Восстановить “второстепенные” функции.

## Входные данные
- Последний успешный бэкап Postgres (зашифрованный).
- Бэкап медиа (или S3 versioning).
- Версия релиза (git SHA) и `production.env`.

## Процедуры

### 1) Восстановление БД
- Поднять новый Postgres.
- Восстановить из последнего бэкапа.
- Прогнать миграции для текущего релиза.
- Проверить базовые API smoke‑checks.

### 2) Восстановление Redis/Celery
- Поднять Redis.
- Запустить worker/beat.
- Проверить readiness `/health/ready/` и выполнение тестовой задачи.

### 3) Восстановление frontend
- Развернуть Next.
- Проверить `/healthz`, `/`, `/catalog`, `/checkout`.

### 4) Возврат платежей/вебхуков
- Проверить доступность webhook endpoint.
- Проверить подписи/ключи провайдера.
- При необходимости временно остановить приём webhook, если есть подозрение на компрометацию.

## Коммуникации
- Канал инцидента.
- Шаблон статуса для клиентов (если понадобится).
- Пост‑мортем в течение 48 часов после восстановления.
